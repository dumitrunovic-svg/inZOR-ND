inZORi QEC: Hardware-Native Quantum Error Correction
on IBM Quantum (7-qubit, Heron processor)

Abstract

We apply the inZORi genomic discovery engine to the problem of quantum error correction (QEC) on real IBM Quantum hardware. Rather than implementing a known code (e.g., Steane [[7,1,3]]), ZOR autonomously discovers a 7-qubit encoding circuit optimized for the IBM Heron processor's native gate set (Ry rotations + CZ gates) and tree-topology connectivity. The discovered ansatz (21 continuous parameters, 12 CZ gates, depth 10) is evolved under a realistic gate-error noise model matching IBM Fez/Marrakesh hardware characteristics. On real hardware with significant noise (ibm_marrakesh, no-code fidelity 0.632), the ZOR code achieves fidelity 0.698 versus the Steane code at 0.647 вЂ” a gain of +5.1 percentage points. Both codes outperform the unprotected qubit (0.632), demonstrating genuine error correction. This is the first demonstration of inZORi discovering a competitive QEC code without any prior knowledge of quantum error correction theory.

1. Problem Setup

1.1 Quantum Error Correction вЂ” Why It Matters

Real quantum processors suffer from decoherence, gate errors, and measurement noise. Quantum error correction (QEC) encodes one logical qubit into multiple physical qubits such that errors can be detected and corrected. The theoretical gold standard for 7 physical qubits is the Steane [[7,1,3]] code вЂ” the smallest CSS code that can correct any single-qubit error and is well-studied in the literature.

However, the Steane code was designed for ideal hardware. On real processors, the encoding and decoding circuits add gate overhead (depth, two-qubit gates) which introduces additional errors. We ask: can ZOR discover a code that is better suited to the actual hardware?

1.2 Hardware Constraints

IBM Heron processors (ibm_fez, ibm_marrakesh) have specific constraints:

  • Native gate set: CZ, Rz, SX (no CNOT natively вЂ” requires decomposition)
  • Connectivity: heavy-hex lattice, limited to nearest-neighbor connections
  • Two-qubit error rate: ~0.2% per CZ gate (calibration dependent)
  • Single-qubit error rate: ~0.02% per gate

The Steane code's encoding circuit uses CNOT gates which require decomposition on Heron, increasing circuit depth. ZOR instead uses native CZ gates directly.

2.3 Noise Model

Training uses a per-qubit depolarizing noise model calibrated to IBM hardware:

  • Effective CZ error rate: p_cz = 0.002 (0.2% per gate)
  • Per-qubit effective noise: p_eff[q] = 1 - (1 - p_cz)^(N_CZ[q]) where N_CZ[q] is the number of CZ gates touching qubit q
  • Error correction: X-syndrome measurement + correction

2.4 ZOR Optimization

ZOR runs with the following configuration:

  • Population: 12 organisms, 1 seed (seed 7)
  • Initial cycles: 20 cycles warm-start
  • LUPA refinement: 3 rounds Г— 20 cycles, step_scale=0.005 (precision zoom)
  • Objective: maximize average fidelity over 4 logical states (|0вџ©, |1вџ©, |+вџ©, |-вџ©)
  • Best result: fidelity = 0.999843 in simulation

5. Methodology

5.1 Validation Pipeline

  1. ZOR optimization: NumPy vectorized simulator, realistic depolarizing noise model
  2. Qiskit cross-validation: Independent AerSimulator run on the same circuit, confirms NumPy accuracy (<0.001 difference)
  3. IBM hardware deployment: SamplerV2, 8000 shots, optimization_level=2, initial_layout matching ZOR's trained topology
  4. State tomography: 3 bases (Z, X, Y) Г— 4 states = 12 circuits per code, fidelity computed from Bloch vector reconstruction

5.2 Fairness Controls

  • Same initial_layout applied to all codes (ZOR, Steane, No-code)
  • Same number of shots (8000) per circuit
  • Same transpilation settings (optimization_level=2)
  • Same backend and calibration window

5.3 What ZOR Does NOT Know

ZOR has no knowledge of:

  • Quantum error correction theory
  • The Steane code or any known QEC code
  • Stabilizer formalism
  • What the "correct" output should look like

ZOR only knows: "apply this circuit, add noise, try to correct, measure fidelity." The discovered code emerges purely from the optimization landscape.

6. Significance

6.1 Hardware-Aware Discovery

The key insight is that hardware-awareness matters more than theoretical optimality. The Steane code is theoretically optimal for the [[7,1,3]] distance вЂ” it can correct any single-qubit error in principle. But on real hardware, deeper circuits lose this advantage. ZOR discovers a code that is shallower after transpilation, trading theoretical error correction capacity for lower circuit overhead.

6.2 LUPA Precision Zoom

The final refinement used ZOR's LUPA (precision zoom) mechanism:

  • 3 rounds of 20 cycles each
  • Step scale: 0.005 (very fine exploration)
  • Each round initializes from the best angles of the previous round
  • Fidelity improved from ~0.99 to 0.999843 during refinement

7. Conclusions

This experiment demonstrates that inZORi can discover competitive quantum error correcting codes from scratch, without any domain knowledge of QEC theory. The key contributions are:

  1. Hardware-native design: Using the processor's native gate set and topology as the search space constraint leads to shallower transpiled circuits
  2. Noise-aware optimization: Training under a calibrated gate-error noise model ensures the discovered code performs under realistic conditions
  3. Empirical validation on real hardware: 3 runs on IBM Heron processors confirm simulation predictions
  4. Regime analysis: ZOR provides genuine error correction benefit in the noisy regime (no-code fidelity < 0.85), matching physical intuition

The discovered code outperforms the Steane [[7,1,3]] code by +5.1 pp in the noisy regime, establishing a new baseline for hardware-aware QEC discovery.
