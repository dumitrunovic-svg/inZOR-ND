Abstract We apply the inZOR-ND bio-adaptive genomic discovery engine to the problem of quantum error correction (QEC) on real IBM Quantum hardware, without any prior knowledge of QEC theory. inZOR-ND autonomously discovers a 7-qubit encoding circuit optimized for the IBM Heron processor's native gate set (Ry + CZ) and tree-topology connectivity — 21 continuous parameters evolved under a calibrated gate-error noise model (p_cz = 0.002 depolarizing per CZ gate). The key contribution is hardware-awareness: native CZ gates transpile to depth ~53 on Heron processors, versus ~110 for the Steane [[7,1,3]] code's CNOT-based encoding (2.2× shallower). On real IBM Marrakesh hardware in a noisy operating regime (no-code fidelity 0.632), the inZOR-ND code achieves fidelity 0.698 versus Steane at 0.647, a gain of +5.1 percentage points. Both codes outperform the unprotected qubit (0.632), confirming genuine quantum error correction. LUPA precision zoom refines simulation fidelity from ~0.97 to 0.999843 in 3 rounds × 20 cycles (~5 minutes total). Independent Qiskit AerSimulator cross-validation confirms NumPy simulator accuracy to <0.001. Validated across 3 IBM Heron hardware runs (ibm_fez × 2, ibm_marrakesh × 1).

1. Introduction

Quantum error correction (QEC) is essential for fault-tolerant quantum computing. Physical qubits suffer from decoherence, gate errors, and measurement noise. QEC encodes one logical qubit into multiple physical qubits to enable error detection and correction. The Steane [[7,1,3]] code is the theoretical benchmark for 7 physical qubits — the smallest CSS code correcting any single-qubit error.

However, theoretical optimality does not guarantee hardware optimality. IBM Heron processors (ibm_fez, ibm_marrakesh) impose specific constraints: (a) heavy-hex lattice connectivity (nearest-neighbor only), (b) native gate set CZ/Rz/SX (CNOT requires decomposition), and (c) two-qubit error rate ~0.2% per CZ gate. CNOT decomposition on Heron requires additional single-qubit rotations, roughly doubling circuit depth and introducing proportionally more errors.

We ask: can the inZOR-ND genomic optimization engine discover a QEC code better suited to the actual hardware, without being given any knowledge of QEC theory? The answer is yes — and the mechanism is simple: by restricting the search space to native IBM gates, inZOR-ND naturally finds shallower circuits.

2. inZOR-ND Framework

inZOR-ND is a bio-adaptive genomic discovery engine operating on populations of organisms with continuous genomes. Key properties:

  • No RL/NN/backpropagation: purely genomic evolution with energy-based selection
  • Genome parameters: each organism encodes risk_sensitivity, step_scale, jump_chance, memory_lr
  • LUPA (precision zoom): multi-round refinement with progressively smaller step_scale
  • Warm start: each LUPA round initializes from the best solution of the previous round
  • Domain-agnostic: same engine used for power systems, astrophysics, fusion plasma, refraction emergence

The same inZOR-ND engine, with zero domain-specific modifications, was applied to the 21-dimensional QEC optimization landscape. The engine only receives fidelity scores from the noise simulator — no quantum mechanics information is encoded.

3. Hardware-Native Circuit Ansatz

3.1 IBM Heron Tree Topology

We use a 7-qubit subtree of the IBM Fez/Marrakesh heavy-hex lattice:

q0 (phys 3) / | \ q1 q2 q3 (2) (4) (16) | | | q4 q5 q6 (1) (5) (23) CZ pairs: (0,1), (0,2), (0,3), (1,4), (2,5), (3,6) — all native IBM Heron edges

3.2 Ansatz Structure

The search space is defined by 3 alternating layers of Ry rotations and 2 layers of CZ gates:

Layer 0: Ry(θ₀–θ₆) [7 params] CZ layer 1: CZ(0,1), CZ(0,2), CZ(0,3), CZ(1,4), CZ(2,5), CZ(3,6) [6 native CZ] Layer 1: Ry(θ₇–θ₁₃) [7 params] CZ layer 2: same 6 CZ pairs Layer 2: Ry(θ₁₄–θ₂₀) [7 params] Total: 21 continuous parameters, 12 CZ gates, circuit depth 10

3.3 Noise Model

Per-qubit depolarizing noise calibrated to IBM hardware. For qubit q with N_CZ(q) CZ gates:

p_eff[q] = 1 - (1 - p_cz)^N_CZ(q), p_cz = 0.002 (0.2% per CZ gate) N_CZ per qubit: q0=4, q1=2, q2=2, q3=2, q4=1, q5=1, q6=1 (total over 2 CZ layers)

After encoding + noise channel, X-syndrome measurement (8 projectors: I⊗⁷ + 7 single-qubit X errors) and correction are applied.

6.1 Noise Regime Analysis

QEC codes only benefit when hardware noise is significant enough that error correction gain exceeds circuit overhead. The ibm_fez runs captured a well-calibrated state (no-code ~0.993) where a 7-qubit QEC circuit adds more noise than it corrects. The ibm_marrakesh run encountered genuine noise (no-code 0.632) where QEC is meaningful.

8. Circuit Depth Comparison

inZOR-ND HW-Native331012 CZ (native)~5312 CZ
Steane [[7,1,3]]1486 CNOT*~110~12 CZ*

*CNOT → Rz + CZ + Rz on Heron. Steane's 6 CNOT require 2× more transpiled depth. Despite more native gates, inZOR-ND's circuit is 2.2× shallower after transpilation.

This is the core design advantage: by using CZ gates natively, inZOR-ND avoids all CNOT decomposition overhead. The extra raw gates (33 vs 14) are single-qubit Ry rotations, which have ~10× lower error rate than two-qubit gates and negligible depth impact.

11. Role of inZOR-ND

The entire discovery was powered by inZOR-ND. The engine has zero domain-specific modifications for quantum computing:

  • Input to inZOR-ND: 21 real-valued parameters θ ∈ ℝ²¹
  • Fitness function: average fidelity over 4 states under noise model (scalar → maximize)
  • inZOR-ND's role: find θ* that maximizes fidelity through genomic evolution
  • What inZOR-ND does NOT know: QEC theory, Steane code, stabilizer formalism, quantum gates semantics

The same engine was previously used for: PFΔ power systems (streaming Newton-Raphson optimization), BAWS thermal management, renewable energy study, fusion disruption proximity (plasma current law discovery), TESS exoplanet classification, refraction emergence, social density dynamics. Application to quantum hardware is a new domain validation.

12. Limitations

  • Single seed (seed 7) for final optimization — broader sweep may find better codes
  • Only X-syndrome correction; Y/Z corrections not explicitly handled
  • Hardware results highly sensitive to calibration — 3 runs is small sample
  • The discovered code has not been formally characterized (distance, stabilizer group)
  • Benefit only demonstrated in noisy regime; low-noise hardware does not benefit from QEC

13. Conclusions

inZOR-ND demonstrates hardware-native quantum error correction without domain knowledge:

inZOR-ND: 0.698 > Steane [[7,1,3]]: 0.647 > No-code: 0.632
+5.1 pp vs Steane · +6.7 pp vs unprotected · Genuine QEC confirmed (ibm_marrakesh, noisy regime)

The key insight: hardware-awareness outperforms theoretical optimality in practice. The Steane code is theoretically distance-3 but transpiles to 2.2× deeper circuits on Heron. inZOR-ND discovers a code that is shallower by design — not because it knows QEC theory, but because it is constrained to native gates and optimized under realistic noise.

LUPA precision zoom achieves simulation fidelity 0.999843 from 80 total cycles (~5 minutes), demonstrating efficient high-precision convergence in 21-dimensional continuous landscapes without gradient information.
